1. System Overview

Emuria Regenerative Property Care is a full-stack Laravel web application that manages the complete lifecycle of property health assessments, remediation agreements, and work scheduling for Etogo's property stewardship service.

2. Complete End-to-End Workflow

The system processes each property through nine sequential phases. Each phase must be completed before the next becomes available.

Phase Who Acts What Happens Key Fields Set
Phase 1
Property Setup		 Client Client registers their property with address, type (residential / commercial / mixed-use), unit count, square footage. number_of_units, residential_units, type, square_footage_interior
Phase 2
Inspection Fee Payment		 Client Client pays the $299 inspection fee via Stripe Checkout Session. System verifies payment_status === 'paid' directly with Stripe API before marking as paid — no trust in URL parameters alone. inspection_fee_status = 'paid', inspection_fee_paid_at
Phase 3
Inspection Creation (Step 1)		 Admin / Project Manager Admin creates the inspection record via the Step 1 form. Assigns inspector, sets scheduled date, status. Property snapshot fields are captured at this point (owner name, address, type etc.). status = 'scheduled', owner_name, property_address_snapshot, property_type_snapshot, all snapshot fields
Phase 4
PHAR Data Entry (Step 2)		 Admin / Inspector Admin fills in the PHAR (Property Health Assessment Report) form. Records findings per system/subsystem with labour hours and materials. BDC travel inputs (distance, time) are entered here. Pricing is calculated but the assessment is NOT completed yet — admin can review before committing. findings, bdc_distance_km, bdc_time_minutes, all pricing fields via MergeBridgeCalculator
Phase 5
Preview & Complete Assessment		 Admin / Project Manager After PHAR data is saved, the "Pricing Ready" card on the form shows a summary. Admin can preview the client report and contract draft before finalising. Two preview buttons open the views with an "ADMIN PREVIEW" banner. When satisfied, admin clicks "Complete Assessment" to seal the inspection. status = 'completed', completed_date
Phase 6
Client Views Report & Signs Agreement		 Client Client sees the completed inspection report — pricing breakdown, findings, payment options. Client goes to the Agreement page, reviews the contract (with start/end dates, work hours, visit schedule), then digitally signs by typing their full name and ticking the acknowledgment checkbox. Client signs FIRST. approved_by_client = true, client_approved_at, client_full_name, client_signature
Phase 7
Work Payment		 Client Client chooses payment plan — "Pay in Full" (single charge) or "Pay Per Visit" (charged once per visit, N visits = bdc_visits_per_year). First payment is processed via Stripe PaymentIntent. Status verified with Stripe before recording. work_payment_status = 'paid', payment_plan, installment_months, installments_paid = 1, arp_total_locked
Phase 8
Etogo Countersignature		 Admin / Project Manager After client has signed AND payment is confirmed, an authorized Etogo representative countersigns from the admin inspection show page. This fully executes the agreement. The Work Visit Schedule input form then becomes available. etogo_signed_by, etogo_signed_at
Phase 9
Work Scheduling		 Admin / Project Manager Admin enters the specific visit dates (N = bdc_visits_per_year). Only weekdays Mon–Fri are accepted; Saturdays and Sundays are rejected. Dates are saved as a JSON schedule. The first date becomes the start date; the last becomes the target completion date. Client can view the confirmed visit schedule on their report. work_schedule, planned_start_date, target_completion_date
Note on Scheduling Blockers: The AgreementScheduleService automatically calculates an estimated start date ("next business day") while waiting for client signature and payment. Once the admin manually saves the work schedule (Phase 9), those auto-calculated dates are locked and can no longer be overwritten by the service.

5. Agreement & Signing Flow

Three-Step Execution Process (Defined in Contract Section 14)

The agreement is not legally binding until all three steps are completed. This process is enforced in code — each step is gated by the previous one.

StepActorActionGate (Precondition)
Step 1
Client Signature		 Client Client types full name and ticks acknowledgment on the agreement page. Recorded digitally with timestamp. Inspection must be status = 'completed'
Step 2
Work Payment		 Client Client pays first work payment (full or first visit). Verified via Stripe API. Inspection must be completed
Step 3
Etogo Countersign		 Admin / Project Manager Authorized Etogo representative countersigns from admin inspection show page. Client must have signed AND work payment must be confirmed. Role check: Super Admin / Admin / Project Manager only.
After Step 3: The Work Visit Schedule form becomes available on the admin inspection show page. The agreement is now fully executed and enforceable.

What the Contract Contains

  • Section 1: Job identification, property details, PHAR reference, approval date
  • Section 2: Locked Scope of Work — generated from PHAR findings
  • Section 3: Project Timeline & Work Schedule — start date, completion target, duration, working hours (Mon–Fri, 9 AM–5 PM), confirmed visit dates table
  • Section 4: Pricing & Payment Terms — labour, materials, tools breakdown; deposit/progress/final payment structure
  • Sections 5–13: Change orders, access requirements, quality assurance, warranty, liability, governing law (British Columbia)
  • Section 14: Authorization — explicit three-step signing order with Client (Step 1) and Etogo Representative (Step 3) signature blocks

PDF Download

Client can download the signed agreement as a PDF at any time via the agreement page. The filename is: Client_Agreement_{client-name}_{property-name}.pdf

6. Work Scheduling

Overview

Once the agreement is fully executed (all three signing steps complete), the Etogo admin or project manager schedules the specific work visit dates. This is the only point where start/end dates are manually entered by staff.

Rules

  • All visit dates must be weekdays (Monday to Friday)
  • Weekends (Saturday and Sunday) are rejected by both the server-side validator and the contract text
  • Working hours are 9:00 AM – 5:00 PM (stated in the contract, shown on the client report)
  • Number of visits = bdc_visits_per_year (auto-derived as ceil(total_labour_hours / 8))
  • Duplicate dates are silently deduplicated; dates are auto-sorted ascending

How Admin Enters the Schedule

  1. Admin navigates to the inspection show page (Admin → Inspections → [Inspection] → Show)
  2. After countersigning, a "Work Visit Schedule" card (green) appears automatically
  3. The card shows N date-picker inputs (N = bdc_visits_per_year), pre-filled with suggested consecutive weekdays starting from tomorrow
  4. Admin adjusts dates as needed and clicks "Save Visit Schedule"
  5. If any selected date is a weekend, the form is rejected with a clear error showing which dates are invalid

Client View

The client sees the confirmed visit schedule as a table on their inspection report page. If the agreement has been countersigned but no schedule has been saved yet, a message reads: "Your work visit dates are being finalised by the Etogo team. You will see them here once confirmed."

Date Lock Behaviour

The AgreementScheduleService runs automatically at several points (after sign, after payment, after countersign) to compute an estimated start date. Once the admin saves a manual schedule, these auto-calculated dates are frozen — subsequent refresh() calls will no longer overwrite planned_start_date or target_completion_date.

9. Authorization & Security

Client Inspection Authorization

Every client-facing inspection method begins with a single authorization check via a private helper method:

private function authorizeInspection(Inspection $inspection): void
{
    if (!$inspection->property || (int) $inspection->property->user_id !== (int) Auth::id()) {
        abort(403, 'Unauthorized access to this inspection.');
    }
}

This method is called at the start of all 9 client inspection methods. No inline duplication exists.

Stripe Payment Security

  • Inspection Fee: On the success redirect, session_id query parameter is required. The system calls $stripe->checkout->sessions->retrieve($sessionId) and checks payment_status === 'paid' before updating the database. URL manipulation alone cannot mark a fee as paid.
  • Work Payment: The submitted payment_intent_id is verified by calling $stripe->paymentIntents->retrieve($id) and checking status === 'succeeded' before recording payment.
  • Per-Visit Payment: Same PaymentIntent verification. Additionally gated by payment_plan === 'per_visit' check — clients on a full plan cannot call this endpoint.
